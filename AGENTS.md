# AGENTS.md — Instructions for AI Coding Agents

## Project

Build a local-first CV Tailoring Agent POC.

Stack:

- Tauri
- Next.js
- TypeScript
- Tailwind CSS v4
- Minimal global CSS for tokens, base styles, keyframes, and CV paper rendering
- Zod
- pnpm
- Vitest

## Main objective

The app lets a user paste:

1. A current resume in HTML.
2. A job offer text.

Then it:

- parses the resume
- parses the job offer
- scores compatibility
- identifies strengths, weaknesses, blockers, and missing keywords
- generates an improved resume version
- audits every change
- prevents hallucinated claims
- presents the flow as a chat-driven CV adaptation workspace with a right-side CV preview

## Critical rule

Never invent resume facts.

No invented:

- skills
- companies
- job titles
- metrics
- certifications
- degrees
- tools
- responsibilities
- production experience
- seniority

Only use information supported by the resume or explicitly validated by the user.

## RTK command rule

Use `rtk` for commands when available.

Prefer:

```bash
rtk pnpm test
rtk pnpm lint
rtk pnpm build
rtk git status
rtk git diff
rtk rg "pattern" .
rtk find . -maxdepth 3 -type f
```

Fallback to normal commands if RTK is unavailable.

Avoid huge noisy outputs. Use targeted commands.

## Development rules

- Use official generators where possible.
- Do not manually recreate boilerplate generated by Next.js or Tauri.
- Use Zod schemas for all structured data.
- Keep deterministic scoring separate from LLM rewriting.
- Validate LLM output before using it.
- Add tests for parsing, scoring, tailoring, and audit logic.
- Keep privacy local-first.
- Do not add telemetry.
- Do not commit secrets.
- Do not use `any` unless unavoidable and documented.

## Current UI/product direction

ResumeForge now follows the `ResumeForge-2.zip` mockup direction:

- warm cream + terracotta visual system
- fixed left sidebar
- compact topbar
- setup flow for AI provider and master CV
- chat-centered job-offer flow
- inline diagnostic cards, validation questions, and generation state
- right-side CV preview with Original / Adapted / Diff modes

Do not reintroduce the old tabbed dashboard UI (`Report`, `Improved CV`, `Audit`, `Missing Keywords`) unless the user explicitly asks.

Do not reintroduce `components/ui` or shadcn runtime primitives just for the current UI. The active implementation is Tailwind-first; keep `apps/web/src/app/globals.css` limited to design tokens, base styles, keyframes, scrollbar styling, and CV paper/diff rendering.

## Suggested setup commands

```bash
pnpm create next-app@latest resume-forge \
  --ts \
  --tailwind \
  --eslint \
  --app \
  --src-dir \
  --import-alias "@/*"

cd resume-forge

pnpm add zod cheerio nanoid lucide-react domhandler

pnpm tauri init
```

When checking work:

```bash
rtk pnpm lint
rtk pnpm test
rtk pnpm build
```

## POC features

Must have:

- resume HTML input
- job offer text input
- structured parsing
- compatibility scoring
- recruiter-style report
- improved resume HTML
- audit trail
- local save/load
- simple Tauri desktop shell

Must not start with:

- SaaS backend
- auth
- billing
- remote database
- LinkedIn scraping automation
- browser extension
- complex template marketplace

## Definition of done

The POC is done when:

- `pnpm dev` runs the app
- `pnpm tauri:dev` runs the desktop shell
- the user can analyze one CV against one job offer
- scoring and report work
- improved CV HTML is generated
- audit trail explains changes
- unsupported claims are blocked
- the UI remains visually faithful to the ResumeForge v2 mockup
- tests pass
